How it works

Three ideas, deliberately simple.

Most RL traffic controllers need a GPU and a leap of faith. Traffix was designed backwards from what a traffic engineer would deploy.

01

State reduction

thousands of raw signals → one compact state

Instead of feeding the agent raw camera or detector streams, Traffix compresses each approach into a small discrete state — queue levels per approach, current phase, demand level. The state space stays small enough to learn fully and to inspect by hand.

02

Tabular Q-learning

a lookup table, not a black box

The policy is a Q-table over that reduced state. Training is cheap, inference is a table lookup, and every decision the controller can ever make can be audited line by line. That's how it outperforms heavier methods without their compute bill.

03

Hybrid safety layer

the agent proposes, the safety layer disposes

A rule-based layer wraps the learned policy: minimum green of 5 s, a must-switch cap of 30 s so no approach starves, and a guaranteed 3 s yellow on every change. Even an untrained agent can't produce an unsafe signal sequence.

Engineering

Built like a product, not a notebook.

Faithful evaluation loop

The demo backend drives SUMO through TraCI with the exact state definition, action set, timing constants and metric equations used in evaluation — no re-implementation drift between paper and product.

Honest by construction

Both controllers always receive the same routes file and the same random seed. Q-table coverage of the visited states is reported live, including when it's low.

Runs anywhere

SUMO installs from pip, the whole stack ships as one Docker image, and the browser only renders frames — so the demo stays smooth on any laptop and deploys to any host.

Configurable, not hard-coded

Every experiment constant — junction id, approach lanes, phase indices, timing, seeds — lives in one config module and can be overridden by environment variables to target a different junction.
